qiankun改造爬坑指南

qiankun简介

官方介绍

一句话总结：qiankun是基于浏览器url实现的，由路径决定加载哪个子应用，子应用可任意技术栈、独立开发、互不影响。

背景

老项目使用Vue2 + ElementUi，随着业务升级，以及内部组件库升级，各模块需要逐步升级为Vue3 + 自研UI库（基于Elment Plus改造的），所以开始了爬坑之旅。

爬过的坑

常见通用问题

大部分qiankun问题可以在官方的常见问题中找到，以下是我在改造过程中遇到的问题，出现频率可能较高，这里单独解释一下

  • 无报错，子应用未成功加载，检查appConfig.json、appConfigDev.json，activeRule配置是否正确，activeRule应避免与entry相同

  • Application died in status LOADING_SOURCE_CODE: You need to export the functional lifecycles in xxx entry

    排查是vueconfig.js中output.libraryTarget未与主应用保持一致，参考问题9，通常配置为umd格式。
    此类问题，排查思路：先检查下子项目的入口文件有没有 export 生命周期函数，再检查下子项目的打包，最后看看请求到的子项目的文件对不对。

  • Application died in status NOT_MOUNTED: Target container with #container not existed while xxx loading!

    报错是因为微应用加载时容器 DOM 不存在，一开始以为qiankun的start时机不正确，改造到vue的mounted钩子函数中依旧报错。后来排查是因为容器 DOM上有v-if控制，这个变量是从vuex取到的，vuex用了”vue-savedata”做持久化，初始化后localStorage的这个变量始终为false，最后将层级设置大一些（MMD: 5），解决问题。

  • 控制台报 Uncaught SyntaxError: Unexpected token ‘<’

    查看public/index.html引用js路径是否正确，规则查看上述改造过程、查看子应用publicPath设置是否正确，如：”/subapp/sub-business”，注意与Nginx配置一致

  • 子应用中获取全局window对象
    例如：用了innerHtml插入dom，如：<button onclick="click"></button>。由于qiankun的沙箱机制，子应用里window都是proxy，即使在代码里写window.click也是定义在当前代理的window对象中，是不会定义到全局window对象的。获取全局window办法：

项目中的一些问题

  • 微应用打包之后 css 中的字体文件和图片加载 404

    • 一张图使用的UI库，onepicture-ui是将组件的图片打包到dist，项目中使用copy-webpack-plugin拷贝到项目路径下来使用。这样做不会经过项目webpack打包处理，未将静态资源加入__webpack_public_path__。解决方案：组件库升级到1.7.7，已将图片打包为base64
    • 传递给egis的图片url无法找到，因为传递给egis的是public下的图片资源，不会经过webpack打包处理，需要手动添加**__webpack_public_path__**前缀，如： cosnt icon = _webpack_public_path_ + “image/mapIcon.png”

  • 重写router的push、replace、resolve方法，使router.path带有子应用前缀（如：sub-app），导致一些业务中逻辑判断失效

    • 控制台报JSON.pase错误，排查到serverconfig.json路径问题， 改造前获取路径为**”serverconfig.json”，这样写请求路径会拼上当前域名，如当前页面url为：http://www.abc.com/a/b**，则请求路为：**http://www.abc.com/a/b/serverconfig.json**。由于让运维配置Nginx，errorpage 404 500 index.html，所以返回了html的内容，所以JSON.parse自然报错。

几个问题的思考与讨论

子应用之间的路由跳转

基座对router.push、replace、resolve进行了重写，只需写路由的path即可跳转对应的子应用，基座通过props传递给子应用，子应用利用props.router的push等方法进行跳转。

#####项目路由只能用history模式吗？

由于 qiankun是通过 location.pathname 值来判断当前应该加载哪个子项目的，所以需要给每个子项目注入不同的路由 path，而 hash 模式子项目路由跳转不改变 path，所以无影响，history 模式子项目路由设置 base 属性即可。

如果主项目使用 hash 模式，那么得用 location.hash 值来判断当前应该加载哪个子项目，并且子项目都得是 hash 模式，还需要给子项目所有的路由都添加一个前缀，子项目的路由跳转如果之前使用的是 path 也需要修改，用 name 跳转则不用。

如果主项目是 hash 模式子项目为 history 模式，那么跳转到子项目之后，无法跳转到另一个 history 模式的子项目，也无法回到主项目的页面。

应用之间的通信

通常子应用之间最好保持独立、业务解耦，但是通信也有以下几种方案：

  • 简单业务下，通过url传参

  • vue的项目，子应用共用相同全局store

  • 子应用间通信较少时，推荐官方提供的应用间通信方式 - Actions 通信

  • 发布/订阅，基座写一个事件总线下发到各个子应用，应用之间不直接相互交互，我都统一去事件总线上注册事件、监听事件，通过这么一个发布订阅模型来做到应用之间的相互通信。
